## 1. Общие сведения

### 1.1. Назначение документа

Настоящее Руководство определяет цели, принципы и организацию процессов безопасной разработки программного обеспечения (далее — РБПО) в [ЗАПОЛНИТЬ: наименование организации] применительно к продукту [ЗАПОЛНИТЬ: наименование СЗИ].

Руководство является главным организационным документом в системе РБПО и основой для сертификации процессов безопасной разработки в соответствии с Приказом ФСТЭК России №240 от 01.12.2023.

### 2.3. Этапы жизненного цикла

Руководство охватывает все этапы жизненного цикла ПО:

- Анализ требований
- Проектирование архитектуры
- Кодирование и сборка
- Тестирование
- Выпуск и доставка
- Эксплуатация и поддержка
- Вывод из эксплуатации

> **Важно:** Руководство утверждено руководителем организации, издано и доведено до всех работников, имеющих отношение к безопасной разработке ПО (п. 5.1 Приказа ФСТЭК №240).

## 4. Термины и определения

| Термин | Определение |
|--------|-------------|
| **РБПО** | Разработка безопасного программного обеспечения — совокупность процессов, направленных на создание ПО, устойчивого к угрозам безопасности информации |
| **СЗИ** | Средство защиты информации — программное, техническое или криптографическое средство, предназначенное для защиты информации (ГОСТ Р 50922-2006) |
| **Недостаток** | Дефект, ошибка или уязвимость в ПО, обнаруженная в ходе разработки или эксплуатации |
| **Поверхность атаки** | Совокупность точек взаимодействия ПО с внешними субъектами и объектами, через которые возможно несанкционированное воздействие |
| **SAST** | Static Application Security Testing — статический анализ исходного кода |
| **DAST** | Dynamic Application Security Testing — динамический анализ кода при выполнении |
| **SCA** | Software Composition Analysis — композиционный анализ сторонних компонентов |
| **SBOM** | Software Bill of Materials — перечень компонентов ПО |

## 5. Цели безопасной разработки

### 5.1. Стратегические цели

1. Обеспечение соответствия продукта [ЗАПОЛНИТЬ: наименование СЗИ] требованиям ГОСТ Р 56939-2024 и Приказа ФСТЭК №240.
2. Минимизация рисков внедрения уязвимостей на всех этапах жизненного цикла ПО.
3. Получение и поддержание сертификата соответствия процессов РБПО.

### 5.2. Операционные цели

1. Внедрение всех применимых процессов РБПО из ГОСТ Р 56939-2024.
2. Автоматизация контролей безопасности в CI/CD-конвейере.
3. Обеспечение прослеживаемости от требований безопасности до результатов тестирования.
4. Систематическое выявление и устранение уязвимостей до выпуска релизов.
5. Непрерывное повышение квалификации сотрудников в области безопасной разработки.

### 5.3. Измеримые показатели

| Показатель | Целевое значение | Периодичность |
|------------|------------------|---------------|
| Доля устранённых критических уязвимостей перед релизом | 100% | Каждый релиз |
| Покрытие SAST-анализом | 100% кодовой базы | Каждый коммит |
| Время устранения критических уязвимостей | [ЗАПОЛНИТЬ: SLA, например ≤ 3 рабочих дней] | Непрерывно |
| Прохождение обучения сотрудниками | 100% | Ежегодно |
| Актуальность SBOM | Обновляется при каждом изменении зависимостей | Непрерывно |

## 10. Правила внутренних проверок

### 10.1. Цели проверок

Внутренние проверки проводятся для:

1. Контроля соблюдения процессов РБПО, описанных в настоящем Руководстве.
2. Оценки эффективности реализованных процессов.
3. Выявления несоответствий и областей для улучшения.
4. Подготовки к внешней сертификации.

### 10.2. Периодичность

| Тип проверки | Периодичность | Ответственный |
|-------------|---------------|---------------|
| Плановая комплексная проверка | [ЗАПОЛНИТЬ: например, 1 раз в 6 месяцев] | Руководитель РБПО |
| Проверка перед релизом | Каждый релиз | Инженер по безопасности |
| Проверка при инциденте | По факту инцидента | Руководитель РБПО |
| Самоконтроль (чеклист) | [ЗАПОЛНИТЬ: например, ежемесячно] | Ответственные за процессы |

### 10.3. Процедура проведения

1. **Планирование** — определение объёма и объектов проверки, формирование чеклиста.
2. **Проведение** — анализ артефактов, интервью с участниками процессов, проверка инструментов.
3. **Фиксация результатов** — протокол проверки с перечнем несоответствий.
4. **Корректирующие действия** — план устранения несоответствий со сроками.
5. **Контроль исполнения** — проверка выполнения корректирующих действий.

### 10.4. Чеклист внутренней проверки

| № | Проверяемый аспект | Критерий |
|---|-------------------|----------|
| 1 | Актуальность Руководства РБПО | Документ пересмотрен и актуален |
| 2 | Наличие артефактов по каждому процессу | Артефакты существуют и не пустые |
| 3 | Работоспособность инструментов SAST/DAST/SCA | Инструменты запускаются, отчёты формируются |
| 4 | Выполнение плана обучения | Все сотрудники прошли обучение по плану |
| 5 | Управление недостатками | Все критические и высокие уязвимости обработаны в срок |
| 6 | Целостность среды сборки | Среда соответствует документации, доступ контролируется |
| 7 | Управление секретами | Секреты не хранятся в коде, ротация выполняется |
| 8 | Актуальность SBOM | SBOM соответствует текущему составу зависимостей |

## 11. Процессы улучшения

### 11.1. Источники данных для улучшения

1. Результаты внутренних проверок (раздел 10).
2. Результаты анализа уязвимостей и инцидентов.
3. Обратная связь от команды разработки.
4. Изменения в нормативной базе (обновления ГОСТ, приказов ФСТЭК).
5. Результаты внешних аудитов и сертификации.

### 11.2. Процедура улучшения

1. **Сбор данных** — агрегация результатов проверок, анализ метрик (раздел 5.3).
2. **Анализ** — выявление системных проблем и корневых причин.
3. **Планирование улучшений** — формирование плана мероприятий с приоритетами и сроками.
4. **Реализация** — внедрение улучшений в процессы.
5. **Оценка результативности** — проверка эффективности внесённых изменений.

### 11.3. Пересмотр Руководства

Настоящее Руководство подлежит пересмотру:

- Не реже **1 раза в год**.
- При существенных изменениях в продукте или процессах.
- При изменении нормативной базы.
- По результатам внутренних или внешних проверок.
