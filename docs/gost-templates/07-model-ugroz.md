## 1. Общие сведения

### 1.1. Назначение документа

Настоящий документ определяет модель угроз безопасности информации и описание поверхности атаки для продукта [ЗАПОЛНИТЬ: наименование СЗИ]. Документ разработан в соответствии с требованиями ГОСТ Р 56939-2024, п. 5.7.

Модель угроз является основой для:

- Определения перечня актуальных угроз безопасности, которым подвержено ПО.
- Описания поверхности атаки — точек взаимодействия ПО с внешними субъектами и объектами.
- Формирования и уточнения требований безопасности (процесс 5.3).
- Определения перечня подсистем, подлежащих углублённому анализу безопасности.
- Обоснования архитектурных решений в области безопасности (процесс 5.6).
- Приоритизации мер защиты и планирования тестирования безопасности.

## 4. Методология моделирования угроз

### 4.1. Общий подход

Моделирование угроз выполняется в следующем порядке:

1. **Описание защищаемого объекта** — определение активов, субъектов доступа и границ системы (раздел 5).
2. **Описание поверхности атаки** — выявление и систематизация точек входа (раздел 6).
3. **Определение модели нарушителя** — описание типов нарушителей, их возможностей и мотивации (раздел 7).
4. **Идентификация угроз** — систематическое выявление угроз методом STRIDE (раздел 4.2).
5. **Оценка рисков** — оценка вероятности и ущерба по модели DREAD (раздел 4.3).
6. **Определение контрмер** — выбор мер защиты для каждой актуальной угрозы (раздел 8).
7. **Соотнесение с ГОСТ Р 58412-2019** — маппинг выявленных угроз на каталог (раздел 9).

### 4.2. Модель STRIDE — идентификация угроз

Для систематической идентификации угроз используется модель STRIDE. Каждый компонент и каждая точка входа анализируются по шести категориям:

| Категория | Расшифровка | Описание | Нарушаемое свойство безопасности |
|-----------|-------------|----------|----------------------------------|
| **S** — Spoofing | Подмена идентичности | Злоумышленник выдаёт себя за другого пользователя, компонент или систему. Примеры: подделка токена аутентификации, подмена IP-адреса, фишинг | Аутентификация |
| **T** — Tampering | Модификация данных | Несанкционированное изменение данных в процессе хранения или передачи. Примеры: модификация записей в БД, подмена параметров запроса, MITM-атака | Целостность |
| **R** — Repudiation | Отказ от авторства | Пользователь отрицает выполнение действия, а система не может это опровергнуть. Примеры: удаление записей журнала, отсутствие аудита операций | Неотказуемость |
| **I** — Information Disclosure | Раскрытие информации | Несанкционированный доступ к конфиденциальным данным. Примеры: утечка персональных данных, раскрытие стека технологий через ошибки, доступ к файлам конфигурации | Конфиденциальность |
| **D** — Denial of Service | Отказ в обслуживании | Нарушение доступности системы или её компонентов. Примеры: DDoS, исчерпание ресурсов, блокировка учётных записей | Доступность |
| **E** — Elevation of Privilege | Повышение привилегий | Получение привилегий, превышающих назначенные. Примеры: эксплуатация уязвимости для получения прав администратора, обход авторизации, IDOR | Авторизация |

#### 4.2.1. Порядок применения STRIDE

Для каждого элемента архитектуры и каждой точки входа выполняется анализ:

1. Определить, какие категории STRIDE применимы к данному элементу.
2. Для каждой применимой категории сформулировать конкретный сценарий угрозы.
3. Зафиксировать угрозу в реестре (раздел 8) с указанием категории STRIDE.
4. Оценить риск по модели DREAD (раздел 4.3).

#### 4.2.2. Соответствие STRIDE типам элементов

| Тип элемента | S | T | R | I | D | E |
|-------------|---|---|---|---|---|---|
| Внешний субъект (пользователь, внешняя система) | + | | + | | | |
| Процесс (компонент, сервис) | + | + | + | + | + | + |
| Хранилище данных (БД, файловая система) | | + | | + | + | |
| Поток данных (сетевое взаимодействие, API-вызов) | | + | | + | + | |

### 4.3. Модель DREAD — оценка рисков

Для приоритизации выявленных угроз используется модель DREAD. Каждая угроза оценивается по пяти критериям по шкале от 1 до 5.

#### 4.3.1. Критерии оценки

| Критерий | Расшифровка | Описание |
|----------|-------------|----------|
| **D** — Damage Potential | Потенциальный ущерб | Насколько серьёзен ущерб при реализации угрозы? |
| **R** — Reproducibility | Воспроизводимость | Насколько легко воспроизвести атаку? |
| **E** — Exploitability | Эксплуатируемость | Какой уровень квалификации и какие ресурсы необходимы для эксплуатации? |
| **A** — Affected Users | Затронутые пользователи | Какая доля пользователей будет затронута? |
| **D** — Discoverability | Обнаруживаемость | Насколько легко обнаружить уязвимость? |

#### 4.3.2. Шкала оценки Damage Potential (Потенциальный ущерб)

| Балл | Описание |
|------|----------|
| 1 | Минимальный ущерб: раскрытие несущественной информации |
| 2 | Незначительный ущерб: временное нарушение работы отдельного компонента |
| 3 | Умеренный ущерб: раскрытие ограниченного объёма чувствительных данных, нарушение работы сервиса |
| 4 | Существенный ущерб: утечка значительного объёма конфиденциальных данных, длительный простой |
| 5 | Критический ущерб: полная компрометация системы, утечка всей базы данных, уничтожение данных |

#### 4.3.3. Шкала оценки Reproducibility (Воспроизводимость)

| Балл | Описание |
|------|----------|
| 1 | Практически невоспроизводимо: требуется сочетание редких условий, состояние гонки |
| 2 | Сложно воспроизвести: требуются специфические условия среды или конфигурации |
| 3 | Умеренно воспроизводимо: требуется определённая последовательность действий |
| 4 | Легко воспроизводимо: атака повторяется стабильно при выполнении нескольких шагов |
| 5 | Всегда воспроизводимо: достаточно одного запроса, атака тривиальна |

#### 4.3.4. Шкала оценки Exploitability (Эксплуатируемость)

| Балл | Описание |
|------|----------|
| 1 | Требуется глубокая экспертиза, собственные инструменты, длительная подготовка |
| 2 | Требуются продвинутые навыки и специализированные инструменты |
| 3 | Требуются базовые навыки в безопасности и общедоступные инструменты |
| 4 | Требуются минимальные технические навыки, доступны готовые эксплойты |
| 5 | Не требуется специальных навыков: эксплуатация через браузер или простой запрос |

#### 4.3.5. Шкала оценки Affected Users (Затронутые пользователи)

| Балл | Описание |
|------|----------|
| 1 | Менее 1% пользователей, единичные случаи |
| 2 | 1–10% пользователей, затронута малая группа |
| 3 | 10–50% пользователей или отдельная категория (например, все администраторы) |
| 4 | 50–80% пользователей |
| 5 | Все пользователи системы (100%) |

#### 4.3.6. Шкала оценки Discoverability (Обнаруживаемость)

| Балл | Описание |
|------|----------|
| 1 | Уязвимость скрыта, требуется доступ к исходному коду и глубокий реверс-инжиниринг |
| 2 | Обнаружение требует целенаправленного анализа с использованием специализированных инструментов |
| 3 | Обнаружение возможно при ручном тестировании безопасности |
| 4 | Обнаружение возможно автоматическими сканерами уязвимостей |
| 5 | Уязвимость очевидна, информация публично доступна (CVE, публикация эксплойта) |

#### 4.3.7. Расчёт уровня риска

Итоговый балл DREAD рассчитывается как среднее арифметическое пяти критериев:

```
DREAD Score = (D + R + E + A + D) / 5
```

Уровни риска по итоговому баллу:

| Итоговый балл | Уровень риска | Описание | Действие |
|---------------|---------------|----------|----------|
| 4.0–5.0 | **Critical** | Критический риск | Немедленное устранение. Блокирует релиз |
| 3.0–3.9 | **High** | Высокий риск | Устранение в ближайшей итерации. Блокирует релиз при отсутствии компенсирующих мер |
| 2.0–2.9 | **Medium** | Средний риск | Устранение планируется в рамках текущего релизного цикла |
| 1.0–1.9 | **Low** | Низкий риск | Устранение по возможности. Включается в бэклог |

#### 4.3.8. Пример оценки

| Угроза | D | R | E | A | D | Итого | Уровень |
|--------|---|---|---|---|---|-------|---------|
| SQL-инъекция в форме поиска | 5 | 5 | 4 | 5 | 4 | 4.6 | Critical |
| CSRF при смене пароля | 3 | 4 | 3 | 3 | 3 | 3.2 | High |
| Раскрытие версии сервера в HTTP-заголовках | 1 | 5 | 5 | 5 | 5 | 4.2 | Critical |
| Brute-force аутентификации | 3 | 4 | 4 | 2 | 4 | 3.4 | High |

> **Примечание.** Оценки в таблице приведены в качестве примера. При заполнении документа необходимо провести оценку для каждой угрозы из реестра (раздел 8).

|---|---|---|---|---|---|
| **Вероятность: 5 (очень высокая)** | Medium | Medium | High | Critical | Critical |
| **Вероятность: 4 (высокая)** | Low | Medium | High | High | Critical |
| **Вероятность: 3 (средняя)** | Low | Medium | Medium | High | High |
| **Вероятность: 2 (низкая)** | Low | Low | Medium | Medium | High |
| **Вероятность: 1 (очень низкая)** | Low | Low | Low | Medium | Medium |

## 10. Перечень подсистем для углублённого анализа безопасности

> Требование ГОСТ Р 56939-2024, п. 5.7.2 — определить перечень подсистем, подлежащих углублённому анализу.

Приоритет определяется на основании количества и критичности связанных угроз, доступности подсистемы из недоверенных зон и критичности обрабатываемых данных.

## 11. Порядок актуализации модели угроз

> Требование ГОСТ Р 56939-2024, п. 5.7.2 — уточнять модель угроз при изменении архитектуры.

### 11.1. Триггеры актуализации

Модель угроз подлежит пересмотру и актуализации при наступлении следующих событий:

| № | Триггер | Описание | Ответственный |
|---|---------|----------|---------------|
| 1 | Изменение архитектуры | Добавление новых компонентов, изменение взаимодействия между существующими, смена технологического стека | [ЗАПОЛНИТЬ: архитектор безопасности] |
| 2 | Добавление новой функциональности | Новые API-эндпоинты, новые интерфейсы пользователя, новые интеграции с внешними системами | [ЗАПОЛНИТЬ: архитектор безопасности / инженер по безопасности] |
| 3 | Обнаружение новых уязвимостей | Выявление уязвимостей в продукте или используемых зависимостях, публикация новых CVE для компонентов системы | [ЗАПОЛНИТЬ: инженер по безопасности] |
| 4 | Инцидент безопасности | Реализация угрозы или попытка атаки — пересмотр применимости и полноты модели | [ЗАПОЛНИТЬ: руководитель РБПО] |
| 5 | Изменение среды эксплуатации | Миграция в облако, смена сетевой топологии, изменение политик безопасности инфраструктуры | [ЗАПОЛНИТЬ: архитектор безопасности / DevOps] |
| 6 | Изменение нормативных требований | Обновление ГОСТ, приказов ФСТЭК, отраслевых стандартов | [ЗАПОЛНИТЬ: руководитель РБПО] |
| 7 | Плановый пересмотр | Регулярный пересмотр вне зависимости от наличия изменений | [ЗАПОЛНИТЬ: архитектор безопасности] |

### 11.2. Периодичность планового пересмотра

| Мероприятие | Периодичность | Ответственный |
|-------------|---------------|---------------|
| Полный пересмотр модели угроз | [ЗАПОЛНИТЬ: не реже 1 раза в год] | [ЗАПОЛНИТЬ: архитектор безопасности] |
| Проверка актуальности поверхности атаки | [ЗАПОЛНИТЬ: при каждом мажорном релизе] | [ЗАПОЛНИТЬ: инженер по безопасности] |
| Пересмотр реестра угроз | [ЗАПОЛНИТЬ: ежеквартально] | [ЗАПОЛНИТЬ: инженер по безопасности] |
| Обновление маппинга на ГОСТ Р 58412-2019 | [ЗАПОЛНИТЬ: при каждом пересмотре реестра] | [ЗАПОЛНИТЬ: инженер по безопасности] |

### 11.3. Процедура актуализации

1. **Инициация** — фиксация триггера актуализации, создание задачи в трекере.
2. **Анализ изменений** — определение затронутых разделов модели (поверхность атаки, реестр угроз, модель нарушителя).
3. **Обновление модели** — внесение изменений в соответствующие разделы документа.
4. **Повторная оценка рисков** — пересчёт оценок DREAD для затронутых и новых угроз.
5. **Обновление перечня подсистем** — пересмотр приоритетов углублённого анализа (раздел 10).
6. **Согласование и утверждение** — согласование обновлённой модели с [ЗАПОЛНИТЬ: роли согласующих].
7. **Уведомление заинтересованных сторон** — информирование команды разработки об изменениях в модели угроз.
8. **Фиксация в истории изменений** — запись в раздел 12 настоящего документа.
